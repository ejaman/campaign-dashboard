## 1. 전략적 컨텍스트 엔지니어링 (Preparation)

AI와의 협업 구조를 **CLAUDE.md**, **TODO.md**, **개발 워크플로우** 세 축으로 설계했습니다. CLAUDE.md는 "어떻게 작성할지"를 고정하고, TODO.md는 "무엇을 만들지"를 명확히 하며, 개발 워크플로우는 이 두 파일을 매 feature 개발 사이클에 일관되게 활용하는 반복 구조입니다.

---

### CLAUDE.md — 프로젝트 규칙 주입

프로젝트 시작 전 Claude Code가 일관된 코드를 생성하도록 CLAUDE.md에 가이드라인을 선구축했습니다.

다음은 가이드라인 예시입니다.

```
- 기술 스택과 버전 (Next.js 16, React 19, Tailwind v4, pnpm 등)
- Layered + Feature 혼합형 폴더 구조와 각 디렉토리의 역할
- 개발 규칙 — 코드 작성(TypeScript 제약, 에러 핸들링) / 데이터 레이어(상태 분리 원칙) / 컴포넌트 레이어(렌더링 성능, UX) / 협업(커밋 컨벤션)
- 데이터 정규화 규칙과 파생 지표 계산 공식을 명시해 AI가 임의로 다른 방식으로 계산하지 않도록 고정
```

파편화된 규칙(절대 규칙, 컴포넌트 작성 원칙, 코딩 컨벤션 등 6개 섹션)을 플로우 순서로 통합하도록 요청했고, AI가 제안한 구조를 수용했습니다. 단, "이미지 최적화(packages/ 폴더 예외)", "Core Web Vitals 90+", "CI/CD(Vercel, Supabase)" 등 이 프로젝트와 무관한 내용은 직접 지시해 제거했습니다. AI는 제거 여부를 판단하지 않고 나열했기 때문에, 프로젝트 맥락을 아는 개발자가 범위를 결정했습니다.

과제 평가 항목(아키텍처 20점, 코드 품질 15점, AI 활용 20점 등)을 제시하고 CLAUDE.md에서 빠진 규칙을 보완하도록 요청해, 에러 핸들링·로딩 상태·UX & 접근성·반응형 규칙을 추가했습니다.

#### 점진적 보완

초기 작성 이후 CLAUDE.md를 고정된 문서로 두지 않고, 개발하면서 동일한 지시나 수정이 반복될 때마다 즉시 규칙으로 추가했습니다.

- **반복 수정 → 규칙화**: 접근성 오류(`htmlFor` 미연결, `aria-label` 중복), Controller 패턴, defaultValues 위치 같은 결정이 세션마다 달라지는 경우 즉시 반영
- **`shared/` 레이어 기준 정의**: `AsyncBoundary` 배치 위치를 AI에게 되물었고, "여러 feature에서 재사용되는 UI 조립 틀"이라는 기준을 직접 정의해 추가

이후 세션에서 동일한 실수가 재발하지 않았으며, AI가 새 컴포넌트를 생성할 때 폴더 기준을 자체적으로 판단하게 되었습니다.

---

### TODO.md — 작업 단위 관리

요구사항 원문을 붙여넣고, feature 브랜치 단위로 체크리스트를 구조화하도록 요청했습니다. AI가 생성한 TODO에 평가 지표 표(배점 포함)를 직접 추가했습니다. 개발 중 우선순위 판단에 참고하기 위해서입니다.

TODO.md는 단순한 작업 목록이 아니라 AI와의 협업 인터페이스로 활용했습니다. 매 세션 시작 시 AI가 TODO.md를 읽고 미완료 항목을 기준으로 구현 계획을 수립하도록 CLAUDE.md에 워크플로우 규칙을 명시했으며, 작업 완료 시 해당 항목을 즉시 체크하도록 했습니다.

---

### 개발 워크플로우 — 매 feature 반복 사이클

feature 단위 개발 시 아래 흐름을 반복했습니다. CLAUDE.md가 "어떻게"를, TODO.md가 "무엇을"을 제공하는 구조 위에서 작동합니다.

1. **TODO 기준 구현 계획 수립**
   AI가 TODO.md의 미완료 항목을 읽고, 필요한 파일·컴포넌트·데이터 흐름을 텍스트 계획으로 먼저 제시합니다. 이 단계에서 불필요한 추상화나 범위 초과를 사전에 걸러냅니다.

2. **계획 검토 후 진행 승인**
   제시된 계획을 직접 검토하고 방향이 맞으면 구현을 승인합니다. 코드 생성 전 계획을 텍스트로 먼저 받는 것이 핵심으로, 방향이 잘못된 경우 코드 없이 수정할 수 있습니다.

3. **TODO 항목 단위 커밋**
   구현 완료 후 TODO의 체크리스트 항목 하나하나를 기준으로 커밋을 나눠 남깁니다. 커밋 메시지를 확인하고 최종 승인합니다.

4. **PR 코드 리뷰 및 피드백 반영**
   PR을 열면 Gemini Code Assist가 자동으로 코드 리뷰를 진행합니다. 리뷰 결과를 직접 읽고 아래 기준으로 선별해 반영합니다.
   - **우선순위 기준** — 리뷰어가 medium/high로 표시한 항목 우선 검토
   - **컨벤션 부합 여부** — CLAUDE.md에 정의된 규칙과 일치하는 피드백만 반영
   - **가독성 저하 항목 선처리** — 코드 품질·가독성 관련 피드백은 기능 피드백보다 먼저 처리

## 2. 구현 단계에서의 비판적 의사결정

### AI 제안에 다른 구조를 직접 제안한 사례

AI의 초안을 그대로 수용하지 않고, 더 나은 아키텍처 방향을 직접 제안한 사례들입니다.

1. `ChartShell` 공통 래퍼 추출 + `shared/` 레이어 도입
   로딩/에러 상태 처리를 매 차트마다 반복하지 않도록, Suspense + ErrorBoundary를 감싼 공통 래퍼로 추출할 것을 직접 제안했습니다. 이후 추가될 차트(플랫폼별, 랭킹)도 동일 패턴으로 재사용 가능하도록 설계했습니다. 이 과정에서 "여러 feature에서 재사용되는 UI 조립 틀"이라는 기준을 직접 정의해 `shared/` 레이어로 분리했습니다. 이 기준은 CLAUDE.md에 반영해 이후 세션에서도 일관되게 적용했습니다.

2. 타입 분리 구조 요구
   AI가 단일 타입으로 설계했으나, 데이터 정규화 단계를 명확히 하기 위해서 API 응답 원본(`RawCampaign`)과 정규화된 도메인 타입(`Campaign`)을 명확히 분리하도록 직접 지시했습니다. 이때 `Campaign`을 처음부터 새로 작성하지 않고 `Omit<RawCampaign, 'platform' | 'status' | ...>`으로 원본 타입을 확장해 중복을 줄였습니다.

3. 컴파운드 컴포넌트 패턴 요구
   AI 초안이 단순 prop-drilling 방식(`title`, `onClose`를 Modal props로 전달)이었습니다. 모달이 `<form>` 태그를 내부적으로 제어해야 하는 구조에서 prop-drilling은 레이아웃을 고정시키고, 이후 다른 종류의 모달이 추가될 때 확장성을 제한합니다.
   이를 `Modal.Header` / `Modal.Content` / `Modal.Footer` 컴파운드 패턴으로 재작성하도록 직접 지시했습니다.

4. FormItem 아키텍처 제안
   AI가 `FormField` / `FormSelect` / `FormDateField` 개별 wrapper를 만들었으나, 폼 필드가 늘어날수록 wrapper도 늘어나는 구조적 비효율을 파악했습니다.
   label+error+suffix를 하나의 `FormItem`이 담당하고 `React.cloneElement`로 자식 input에 `id`를 주입하는 방식으로 재설계하도록 제안했습니다. 이를 통해 `Input`, `Select`, `DateInput` 각각을 도메인 무관한 독립 컴포넌트로 유지하면서 label-input 접근성 연결을 `FormItem`이 일관되게 처리합니다.

5. Zustand → URL Query Params 전환
   AI가 Zustand로 초기 구현했으나, URL 기반 상태 관리가 "공유 가능한 URL", "뒤로가기 동작", "새로고침 유지"를 보장한다는 이유로 전환을 직접 제안했습니다.
   기능 완성 후 구조를 다시 뒤집는 결정이었으나 평가 항목(상태관리 10점)과 UX 품질을 고려해 진행했습니다.

6. 전체 선택 = URL 파라미터 없음 패턴
   AI 초안은 `platforms=Google,Meta,Naver`처럼 전체 값을 명시했으나, URL을 불필요하게 길게 만들고 초기값 변경 시 호환성이 없다는 문제를 파악했습니다.
   "전체 선택" 상태를 URL에 기록하지 않고 파라미터 부재로 표현하는 설계를 직접 제안했습니다.

7. `MetricToggle` 범용화
   일별 차트 전용으로 만들어진 AI 결과물에 `options` prop을 추가해 다른 차트에서도 재사용 가능한 범용 컴포넌트로 확장하도록 직접 요청했습니다.

8. 검색 방식 3회 수정
   `useTransition`을 적용해 입력 반응성은 유지하면서 테이블 업데이트를 낮은 우선순위로 처리하는 방식을 채택했습니다. AI 결과물을 그대로 수용하지 않고 UX 관점에서 반복 검토했습니다.

### AI 작업물을 직접 수정한 사례

AI의 초안에서 버그나 UX 문제를 직접 발견하고 수정을 지시한 사례들입니다.

1. `getDefaultDateRange` 함수 스코프 조정 (feat/filter)
   AI가 날짜 기본값을 모듈 스코프 상수로 정의해, 서버 빌드 시 날짜가 고정되는 버그가 발생했습니다. 직접 발견하고 함수 호출 방식으로 교체하도록 지시했습니다.

2. 그래프 반만 보이는 현상 디버깅 (feat/chart-daily)
   클릭수만 활성화될 때 그래프가 반만 보이는 현상을 직접 발견했습니다.
   AI가 조건부 렌더링(`&&`)을 원인으로 지목했으나, 재현 조건(두 지표 동시 활성 → 하나 비활성화)을 직접 좁혀 YAxis 도메인 재계산 애니메이션이 원인임을 확인했습니다.
   필터 변경 시에도 동일 현상이 발생함을 직접 발견해 Recharts의 구조적 한계(데이터 변경 시 entrance 애니메이션 재실행)로 원인을 확정했습니다.

3. 컴포넌트 뎁스 조정 (feat/table)
   AI가 `CampaignTableContainer` 중간 레이어를 추가했으나, 뎁스가 불필요하게 깊다고 판단해 삭제를 지시했습니다.
   `CampaignTableMeta` + `CampaignTableContent`를 `AsyncBoundary` 안에 형제로 배치하는 구조로 변경했습니다. TanStack Query 캐시 덕분에 두 컴포넌트가 동일 쿼리를 각각 호출해도 네트워크 중복이 없음을 확인 후 결정했습니다.

4. 컬럼 고정 너비 (feat/table)
   검색 결과 변화에 따라 테이블 레이아웃이 흔들리는 문제를 직접 발견했습니다. `table-fixed` + 퍼센트 너비 방식으로 수정하도록 지시했습니다.

5. `watch()` → `useWatch()` 교체 (feat/campaign-modal)
   React Compiler가 `watch()`의 반환 함수를 메모이제이션할 수 없다고 경고했습니다. 코드 실행 결과를 직접 확인하고 `useWatch({ control, name })` 방식으로 교체하도록 지시했습니다.

### Gemini 코드 리뷰 반영 사례

**CHART_METRICS 튜플로 통합** (feat/chart-daily PR)

Gemini Code Assist PR 리뷰에서 `METRIC_LABELS`, `METRIC_COLORS`가 별도 Record로 분리되어 있고, `ChartMetric` 타입까지 3곳에 분산된 점을 지적받았습니다. 이를 계기로 단순 수정이 아닌 구조 개선을 직접 제안했습니다.

```ts
export const CHART_METRICS = [
  { key: 'impressions', label: '노출수', color: '#0081cf' },
  ...
] as const

export type ChartMetric = (typeof CHART_METRICS)[number]['key']
```

- **단일 진실 공급원**: 지표 추가 시 배열 한 곳만 수정하면 타입·레이블·색상이 자동 반영됩니다.
- **타입 자동 파생**: `as const` + `typeof`로 타입을 데이터에서 추론해 타입과 데이터 불일치를 원천 차단합니다.
- **순서 보장**: 배열이므로 렌더링 순서가 선언 순서를 따릅니다. Record는 순서를 보장하지 않습니다.
